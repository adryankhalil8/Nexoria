Technical Deep Dive~2 min
6 / 9

3 Hard Problems — Solved

🔐

403 vs 404 — Info Leakage

Returning 403 Forbidden when a user didn't own a resource confirmed the resource existed — enabling ID enumeration.

Before
GET /api/blueprints/42
→ 403 Forbidden

Attacker knows ID 42 exists. Can enumerate IDs to map out all resources.
After
findByIdAndUser() returns Optional → 404 Not Found if not owned. Resource existence is hidden from unauthorized callers.
// BlueprintController.java return repo.findByIdAndUser(id, user) .map(b -> ok(b)) .orElse(notFound().build());
🗄️

MySQL Safe Update Mode

MySQL's sql_safe_updates=1 blocks UPDATE/DELETE statements without a primary key in the WHERE clause — silently breaking PATCH endpoints.

Before
PATCH /api/leads/5
→ MySQL Error 1175:
UPDATE blocked — unsafe update, no PK in WHERE.
After
JPA repositories use PK-based WHERE automatically. Added sql_safe_updates=0 to JDBC URL for local dev environment.
# application-local.yml datasource: url: jdbc:mysql://localhost/nexoria ?sql_safe_updates=0 &allowPublicKeyRetrieval=true
📦

Flyway — Duplicate Column Errors

Adding new columns across 7 migrations without idempotency caused Duplicate column name failures on fresh schema installs.

Before
ALTER TABLE blueprints ADD COLUMN status...
→ Error: Duplicate column if migration partially applied or table rebuilt.
After
Each migration owns its version exclusively. Columns added only in their introducing migration. Strict ordering prevents re-runs. Validated with fresh DB wipe.
-- V4__client_portal_fields.sql ALTER TABLE blueprints ADD COLUMN status VARCHAR(50) DEFAULT 'DRAFT' NOT NULL, ADD COLUMN purchase_event_type VARCHAR(50);
For each problem: state what broke, why it was non-obvious, and what the fix taught you. The 403/404 issue demonstrates security thinking — not just "it failed." The safe updates issue shows you understand how MySQL behaves in real environments. The Flyway issue shows disciplined schema evolution. Frame these as engineering decisions, not accidents.

Auth & Security

JWT Auth Flow — HS256 · JJWT 0.11

1

Client POSTs /api/auth/login with email + password

2

Server returns accessToken (24h) + refreshToken (7d)

3

React stores in sessionStorage · Axios attaches Authorization: Bearer <JWT> on every request

4

JwtAuthFilter validates signature + expiry → sets SecurityContext → controller proceeds

Role-Based Access Control

ADMINUsers · Leads · Blueprints · Scheduling · Support USERClient portal — blueprint, tasks, results, support thread VIEWERRead-only subset of USER routes

Security Headers + Rate Limiting — ADR-003

CSP: default-src 'self' HSTS 1yr + subdomains X-Frame: DENY nosniff Rate limit: 5 req/IP/min on /auth/**

Run Locally

# 1. Start MySQL docker compose up -d db
# 2. Backend (Flyway auto-migrates) cd backend mvn spring-boot:run # → http://localhost:8080 # → /swagger-ui/ available
# 3. Frontend cd frontend npm install && npm run dev # → http://localhost:5173
# 4. Bootstrap first admin POST /api/auth/bootstrap-admin { "secret": "<ADMIN_BOOTSTRAP_SECRET>" }

JWT_EXPIRATION=86400000
ADMIN_BOOTSTRAP_SECRET=first-admin-key

The JWT filter runs on every request — not middleware you opt into, it's part of the filter chain. CSP and HSTS were added deliberately via ADR-003, not retrofitted. Flyway runs on startup automatically so there's no manual SQL to run. sessionStorage was chosen over localStorage to limit the XSS attack window — tokens clear when the tab closes.

Lessons Learned~1 min

What We Learned

🏗️Schema-first thinking pays off. Flyway forced us to plan data transitions before writing code. Every feature required a migration spec first.
🔐Security can't be retrofitted. JWT + RBAC + rate limiting + CSP headers were built from day one via ADR-003 — not added after the fact.
🧩Role boundaries create API clarity. Separating ADMIN, USER, and VIEWER forced explicit contracts — every controller asks "who is allowed here?"
📋Multi-step UX needs lightweight state. SessionStorage was the right call for intake drafts — no server overhead, acceptable loss on tab close for a public flow.
📡SSE over WebSockets for this use case. Support chat is primarily one-directional. Spring's SseEmitter handled it natively without upgrade handshakes.

What's Next

☁️Cloud deployment. Docker Compose → AWS ECS or Fly.io with RDS. Nginx reverse proxy already lives in infra/.
🔄CI/CD pipeline. GitHub Actions → build + test → Docker push → staging deploy. Currently at ~96%; shipping is the remaining 4%.

Questions

Backup Slide
Why Spring Boot over Node.js or FastAPI?
Spring Security is the most mature RBAC + JWT framework available. Java 21 virtual threads give excellent concurrency. JPA + Flyway fit a product with a real, evolving data model — not a scripting job. It's the industry standard in enterprise consulting.
How do you handle token expiry?
Access tokens expire in 24h. A separate 7-day refresh token is issued at login. The client hits POST /api/auth/refresh with the refresh token to get a new access token without re-authenticating.
Why SSE instead of WebSockets for support?
Support chat is primarily one-directional push (admin → client). SSE works over plain HTTP/1.1, no upgrade handshake needed, and Spring's SseEmitter handles it natively. WebSockets would add protocol complexity with zero benefit for this use case.
How would you scale this beyond one server?
Containerize with Docker → ECS or Kubernetes. Move JWT secret + DB credentials to AWS Secrets Manager. Replace Guava in-memory rate limiter with Redis. MySQL → RDS Multi-AZ. CDN in front of the React build. Stateless JWT design makes horizontal scaling clean.
What if two admins update the same blueprint at once?
Currently last-write-wins — no optimistic locking. In production I'd add a @Version field to Blueprint for JPA optimistic locking. The second writer gets a 409 Conflict and must reload before saving.
Why sessionStorage instead of localStorage for the JWT?
sessionStorage clears when the tab closes — this limits the XSS attack window. localStorage persists indefinitely. The app doesn't need "stay logged in" semantics, so sessionStorage is the safer default with no UX trade-off.
